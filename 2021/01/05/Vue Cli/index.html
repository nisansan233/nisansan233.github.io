2.1. 什么是前端路由

什么是前端渲染, 什么是后端渲染?

  • 前端渲染：浏览器中显示的网页中的大部分内容，都是由前端写的js代码在浏览器中执行，最后渲染出来的网页
  • 后端渲染：服务器直接生产渲染好对应的HTML页面, 返回给客户端进行展示.

什么是前后端分离?

  • 随着Ajax的出现, 有了前后端分离的开发模式.
  • 后端只提供API来返回数据, 前端通过Ajax获取数据, 并且可以通过JavaScript将数据渲染到页面中.

什么是前端路由, 什么是后端路由?

  • 前端路由：改变URL，但是页面不进行整体的刷新。

  • 后端路由：后端处理URL和页面之间的映射关系

  • URL的hash

    • URL的hash也就是锚点(#), 本质上是改变window.location的href属性.
    • 我们可以通过直接赋值location.hash来改变href, 但是页面不发生刷新

Vuex

官方解释:VueX是一个专为Vue.js应用程序开发的状态管理模式

什么是状态管理?

  • 其实你可以简单的将其看成把需要多个组件共享的变量全部存储在一个对象里面
  • 然后,将这个对象放在顶层的Vue实例中,让其他组件可以调用
  • 那么多个组件是不是就可以共享这个对象中的所有变量属性呢
  • 多界面状态管理
    • 全局单例模式（大管家）
      • 我们现在要做的就是将共享的状态抽取出来，交给我们的大管家，统一进行管理。
      • 之后，你们每个试图，按照我规定好的规定，进行访问和修改等操作。
      • 这就是Vuex背后的基本思想。

安装

建议创建项目的时候直接选上Vuex这项,在src/store里面的index.js进行配置

  • 做一个简单的小节：
    • 1.提取出一个公共的store对象，用于保存在多个组件中共享的状态
    • 2.将store对象放置在new Vue对象中，这样可以保证在所有的组件中都可以使用到
    • 3.在其他组件中使用store对象中保存的状态即可
      • 通过this.$store.state.属性的方式来访问状态
      • 通过this.$store.commit(‘mutation中方法’)来修改状态
  • 注意事项：
    • 我们通过提交mutation的方式，而非直接改变store.state.count。
    • 这是因为Vuex可以更明确的追踪状态的变化，所以不要直接改变store.state.count的值。

Vuex核心概念

State

  • State单一状态树,Single Source of Truth,也可以翻译成单一数据源
  • 所以Vuex也使用了单一状态树来管理应用层级的全部状态。
  • 单一状态树能够让我们最直接的方式找到某个状态的片段，而且在之后的维护和调试过程中，也可以非常方便的管理和维护。

Getters

  • 类似组件里面的计算属性,在getters里面定义方法
    • 参数默认属性state(必须传入)
  • $store.getters.方法名
    我们可以在Store中定义getters
  • getters默认是不能传递参数的, 如果希望传递参数, 那么只能让getters本身返回另一个函数.

Mutation

  • Mutation状态更新
  • Vuex的store状态(state)的更新唯一方式：提交Mutation
  • 类似组件里面的方法属性,在mutation里定义方法
  • Mutation主要包括两部分：
    • 字符串的事件类型（type）
    • 一个回调函数（handler）,该回调函数的第一个参数就是state。
  • mutation的定义
    • 默认属性state
  • 通过mutation更新
  • 在通过mutation更新数据的时候, 有可能我们希望携带一些额外的参数
    • 参数被称为是mutation的载荷(Payload)
  • Mutation中的代码
  • Mutation提交风格
    • 上面的通过commit进行提交是一种普通的方式
    • vue还提供了另外一种风格,它是包含type属性的对象
    • Mutation中的处理方式是将整个commit的对象作为payload使用, 所以代码没有改变
  • Mutation响应规则
    • Vuex的store中的state是响应式的, 当state中的数据发生改变时, Vue组件会自动更新.
    • 这就要求我们必须遵守一些Vuex对应的规则:
      • 提前在store中初始化好所需的属性
      • 当给state中的对象添加新属性时, 使用下面的方式:
        1. 方式一: 使用Vue.set(obj, ‘newProp’, 123)
        2. 方式二: 用新对象给旧对象重新赋值
  • Mutation常量类型 – 概念
    • 在mutation中, 我们定义了很多事件类型(也就是其中的方法名称).
    • 方法过多, 使用者需要花费大量的经历去记住这些方法, 甚至是多个文件间来回切换, 查看方法名称, 甚至如果不是复制的时候, 可能还会出现写错的情况.
    • 如何避免上述的问题呢?
      • 在各种Flux实现中, 一种很常见的方案就是使用常量替代Mutation事件的类型.
      • 我们可以将这些常量放在一个单独的文件中, 方便管理以及让整个app所有的事件类型一目了然.
    • 具体怎么做呢?
      • 我们可以创建一个文件: mutation-types.js, 并且在其中定义我们的常量.
      • 定义常量时, 我们可以使用ES2015中的风格, 使用一个常量来作为函数的名称.
        如图：
  • Mutation同步函数
    • Mutation中进行异步操作,无法进行响应式Vue Devtools无法进行有效的跟踪,所以通常情况下不要在Mutation中进行异步操作

Action

  • Action类似于Mutation,但是是用来代替Mutation进行异步操作的
    • 默认属性context,等价于store
  • 如果要修改状态,要在Action里使用commit调用Mutation的方法进行状态提交
  • 在Vue组件中, 如果我们调用action中的方法, 那么就需要使用dispatch
  • 同样的, 也是支持传递payload
  • 在Action中, 我们可以将异步操作放在一个Promise中, 并且在成功或者失败后, 调用对应的resolve或reject.

Module

  • 专门划分模块来操作数据
    • Vue使用单一状态树,那么也意味着很多状态都会交给Vuex来管理.
    • 当应用变得非常复杂时,store对象就有可能变得相当臃肿.
    • 为了解决这个问题, Vuex允许我们将store分割成模块(Module), 而每个模块拥有自己的state、mutations、actions、getters等
  • 调用模块里的state时,$store.state.模块名
  • Module局部状态
    • 调用模块里的mutations,依旧是使用this.$store.commit(‘方法名’,payload)
    • 调用模块里的getters,依旧是使用$store.getters.方法名
    • mutation和getters接收的第一个参数是局部状态对象
    • actions的写法呢? 接收一个context参数对象
      • 局部状态通过 context.state 暴露出来，根节点状态则为 context.rootState

        项目结构

  • 当我们的Vuex帮助我们管理过多的内容时, 好的项目结构可以让我们的代码更加清晰.

  • axios实例

    • 为什么要创建axios的实例呢?
      • 当我们从axios模块中导入对象时, 使用的实例是默认的实例.
      • 当给该实例设置一些默认配置时, 这些配置就被固定下来了.
      • 但是后续开发中, 某些配置可能会不太一样.
      • 比如某些请求需要使用特定的baseURL或者timeout或者content-Type等.
      • 这个时候, 我们就可以创建新的实例, 并且传入属于该实例的配置信息.
